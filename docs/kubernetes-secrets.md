# Kubernetes Secrets Flow Guide

This document explains how Kubernetes app secrets are managed in this repo using Vault + External Secrets, with source data defined in Terraform tfvars.

## 1) Source of truth

Secrets used by Kubernetes workloads are sourced from Vault data written by Terraform:

- Terraform stage: `terraform/swarm/vault/config`
- Pipeline entrypoint: `terraform/swarm/vault/config/pipeline/config.sh`
- tfvars source file: `/mnt/eapp/.tfvars/vault/config.tfvars`

The Kubernetes manifests consume those Vault entries through `SecretStore` + `ExternalSecret`.

## 2) End-to-end secret path

```text
/mnt/eapp/.tfvars/vault/config.tfvars
  -> terraform/swarm/vault/config (vault_kv_secret_v2 resources)
    -> Vault KV v2 mount (default: secret)
      -> kubernetes/<app>/secretstore.yaml (Vault auth + connection)
        -> kubernetes/<app>/externalsecret.yaml (remoteRef mapping)
          -> native Kubernetes Secret (target.name)
            -> Deployment/StatefulSet env via secretKeyRef or envFrom
```

## 3) Vault write model from tfvars

`terraform/swarm/vault/config/main.tf` writes data from two maps in tfvars:

- `secrets` for inline key/value fields
- `secret_files` for file-backed fields loaded via `file()`

Terraform flattens entries to Vault paths as:

- `<group>/<secret_name>`

In current usage, group is commonly `k8s`, so keys are typically under:

- `k8s/<secret_name>`

## 4) Kubernetes manifest model

Each app that uses External Secrets usually contains:

- `secretstore.yaml`
- `externalsecret.yaml`

### 4.1 SecretStore contract

`SecretStore` points to Vault and references a namespace-local token secret.

Common fields:

- `spec.provider.vault.server`: Vault URL
- `spec.provider.vault.path`: `secret`
- `spec.provider.vault.version`: `v2`
- `spec.provider.vault.auth.tokenSecretRef.name`: usually `<app>-vault-reader`
- `spec.provider.vault.auth.tokenSecretRef.key`: `VAULT_TOKEN`

### 4.2 ExternalSecret contract

`ExternalSecret` maps one or more Vault fields into a target Kubernetes Secret.

Common fields:

- `spec.secretStoreRef.name`: matches app SecretStore
- `spec.target.name`: generated Kubernetes Secret name (for workloads)
- `spec.data[].secretKey`: key inside generated Kubernetes Secret
- `spec.data[].remoteRef.key`: Vault path (for example `k8s/prowlarr`)
- `spec.data[].remoteRef.property`: field inside that Vault secret

### 4.3 Workload consumption

Deployments then reference the generated Kubernetes Secret with:

- `env[].valueFrom.secretKeyRef`
- or `envFrom.secretRef`

## 5) Argo CD ordering for secrets

Current convention uses sync waves:

- `10`: `SecretStore`
- `15`: `ExternalSecret`
- `20+`: database/app deployments

This ensures secret providers and target secrets are available before workloads start.

## 6) Required bootstrap secret per namespace

Each namespace using Vault-backed External Secrets needs a Kubernetes Secret holding a Vault token:

- name: `<app>-vault-reader`
- key: `VAULT_TOKEN`

Important:

- This token secret is referenced by `SecretStore`.
- It is not generated by `ExternalSecret` itself.
- It must exist in the same namespace as the `SecretStore`.

## 7) Adding secrets for a new app (compliant flow)

1. Add Vault data in `/mnt/eapp/.tfvars/vault/config.tfvars` under `secrets` (and `secret_files` if needed).
2. Apply Vault Terraform stage: `terraform/swarm/vault/config/pipeline/config.sh`.
3. Create app `secretstore.yaml` referencing `<app>-vault-reader` and Vault `secret` mount.
4. Create app `externalsecret.yaml` mapping `remoteRef.key` + `property` to `target.name` + `secretKey`.
5. Consume generated secret in deployment/statefulset env settings.
6. Validate `ExternalSecret` sync and resulting Kubernetes Secret before deploying app changes broadly.

## 8) Naming rules and key-shape rules

- Vault Terraform variable validation allows lowercase alphanumeric, `_`, and `-` in group and secret names.
- `remoteRef.key` in manifests must match the actual Vault path that Terraform writes.
- For tfvars-driven secrets, standardize on `k8s/<secret_name>` style and keep `secret_name` consistent between tfvars and manifests.

If `remoteRef.key` does not match the Vault path exactly, External Secrets will remain out-of-sync and app pods will fail on missing secret keys.

## 9) Refresh and rotation behavior

Most `ExternalSecret` manifests use:

- `refreshInterval: 1m`

Rotation flow:

1. Update tfvars secret values.
2. Re-apply Vault Terraform config stage.
3. Wait for External Secrets refresh cycle.
4. Restart workloads only if app does not hot-reload secret-backed env vars.

## 10) Current exception patterns

- `kubernetes/cross-seed/overlays/bhd/secret.yaml` is a direct Kubernetes `Secret` and does not follow the External Secrets flow.
- `kubernetes/k10/secret-k10-s3-secret.example.yaml` is an example static secret manifest, not Vault-backed automation.

New secret-bearing apps should prefer the Vault + External Secrets pattern described above.
