# Argo CD Rules

This document defines the steady-state Argo CD structure and guardrails for
this repo. Use it for ownership boundaries, source-of-truth locations, and the
GitOps commit/push rule. Use
[docs/workflows/argocd.md](./../workflows/argocd.md) for the operator flow,
[docs/rules/kubernetes.md](./kubernetes.md) for workload layout rules, and
[docs/rules/terraform.md](./terraform.md) for Terraform-managed platform stages.

## Ownership Model

Argo CD in this repo has two control layers:

1. Terraform-managed platform resources under `terraform/cluster/argocd/config`
2. Git-managed Argo CD objects under `kubernetes/argocd-management`

Keep those layers separate.

Use Terraform when the change is about the Argo CD control plane itself or the
Terraform-owned root/addon definitions. Use `kubernetes/argocd-management` when
the change is about repo-managed `AppProject`, `Application`, ingress,
notifications, or other in-cluster Argo CD manifests.

## Bootstrap and Control Points

The repo contains two bootstrap-related entrypoints:

- `kubernetes/bootstrap/argocd-management-app.yaml` is the seed manifest shape
  for the GitOps root app
- `terraform/cluster/argocd/config` is the steady-state owner of the live
  `argocd-management` root application and the `homelab-addons` `ApplicationSet`

The normal control chain is:

1. Terraform ensures the root `argocd-management` application exists
2. that root application points at `kubernetes/argocd-management`
3. repo-managed Argo CD objects under `kubernetes/argocd-management` point at
   workload directories under `kubernetes/`

## Source of Truth Rule

For repo-managed Kubernetes delivery, Git is the source of truth.

Observed repo conventions:

- repo-managed `Application` objects point at `git@github.com:nodadyoushutup/homelab.git`
- repo-managed applications generally use `targetRevision: HEAD`
- automated sync with `prune` and `selfHeal` is the normal mode
- `CreateNamespace=true` and `ServerSideApply=true` are standard sync options
- `SkipDryRunOnMissingResource=true` is used where missing CRDs can be expected

Do not treat a local uncommitted manifest change as a deployable change through
Argo CD. Until the change is committed and pushed, Argo CD cannot reconcile it.

## Commit and Push Rule

When changing a Kubernetes app that is managed by Argo CD:

1. update every repo file that is part of the change
2. stage all files relevant to that Kubernetes/Argo CD change
3. create one clear commit that describes the service and intent
4. push the commit so the tracked Git revision exists remotely
5. let Argo CD autosync that pushed revision

Relevant files commonly include:

- workload manifests under `kubernetes/<app>/`
- overlay files under `kubernetes/<family>/overlays/<instance>/`
- matching `Application` or `AppProject` objects under
  `kubernetes/argocd-management/`
- docs under `docs/` when the pattern changed

Clear commit subjects should follow the repo’s existing style, such as:

- `prowlarr: adjust ingress and secret wiring`
- `qbittorrent: add movie-11 overlay`
- `argocd: document autosync operating model`

Do not rely on an MCP-triggered sync, a manual refresh, or a direct `kubectl apply`
as the way to lock in a steady-state change. Those actions can help with
inspection, debugging, or recovery, but the persistent delivery path is commit
plus push to Git.

## Repository Shape

### Terraform-managed Argo CD layer

`terraform/cluster/argocd/config` owns the Argo CD provider-backed resources
that seed the GitOps control plane.

Current repo examples:

- root `argocd-management` application
- `homelab-addons` `ApplicationSet`

### Git-managed Argo CD layer

`kubernetes/argocd-management` owns repo-managed Argo CD manifests such as:

- per-service `AppProject` objects
- per-service or per-instance `Application` objects
- Argo CD ingress
- notifications config
- Vault-backed Argo CD notification secret wiring
- Argo CD config and terminal/RBAC helper manifests

## Application and Project Rules

For repo-managed app families:

1. create or update the `AppProject` first when the family needs a scoped project
2. keep `AppProject` sync waves lower than their applications
3. keep `Application.spec.source.path` pointed at the exact manifest directory or
   overlay path
4. keep `destination.namespace` aligned with the workload namespace
5. include both SSH and HTTPS forms of the repo URL in `sourceRepos` when
   following the existing project pattern

For multi-instance families such as qBittorrent, the delivery unit is usually:

- one overlay directory
- one Argo CD `Application`
- one destination namespace

## Addons Rule

Some platform addons are not defined as static YAML under
`kubernetes/argocd-management`. They are generated by the Terraform-owned
`homelab-addons` `ApplicationSet`.

When the change belongs to one of those addon entries, update
`terraform/cluster/argocd/config` instead of inventing a second repo-managed
application definition.

## Notifications and Access Rules

Current repo-managed Argo CD support manifests include:

- ingress at `argocd.nodadyoushutup.com`
- Discord notification templates and triggers in
  `kubernetes/argocd-management/notifications-cm.yaml`
- Vault-backed webhook delivery through
  `kubernetes/argocd-management/externalsecret.yaml`

Treat those as Argo CD platform configuration, not workload manifests.

## MCP and Manual Action Rule

The Argo CD MCP server and direct Argo CD or Kubernetes CLI actions are allowed
for:

- inspection
- validation
- targeted refresh
- recovery after failure
- emergency operator intervention

They are not the normal substitute for committing and pushing Kubernetes app
changes.
