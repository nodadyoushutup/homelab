# Requirements Stage

These instructions apply when an issue is in `REQUIREMENTS`, whether the issue
just moved there during the current conversation or the user explicitly asks to
work on an existing issue already in `REQUIREMENTS`.

`REQUIREMENTS` is the stage where the team takes on the idea enough to flesh out
the ticket. Finishing this stage means the issue is ready for technical review
and should move onto the tech lead's plate.

## Requirements Gathering

- Begin requirements gathering from the current Jira issue context, especially
  the baseline description or overview created during `TO DO`.
- The user may provide requirements directly, ask the agent to generate them, or
  provide a partial draft for refinement.
- If the user provides requirements that are light, vague, or likely to miss the
  real shape of the work, say that you have ideas for improving them and offer
  to workshop the requirements.
- When workshopping, interview the user one question at a time. Ask focused
  questions that materially improve the ticket; do not nitpick every small
  detail.
- If the user provided clear enough requirements and there is not much to figure
  out, do not drag them through unnecessary questions. Clean up the requirements
  and proceed.
- If the user asks for autogenerated requirements, act like a strong business
  analyst: infer sensible implementation-facing requirements from the overview,
  issue type, existing description, comments, and known context.
- Autogenerated requirements should be useful direction for whoever takes the
  issue into development, not a speculative implementation plan.

## Requirements Format

- Store requirements in the `Requirements` custom field,
  `customfield_10103`.
- Format the field as a Markdown unordered list.
- Prefix every requirement with a stable `REQ-###` identifier, starting at
  `REQ-001`.
- Keep the list ordered in the most sensible sequence for understanding and
  tackling the work, even though the Markdown structure is unordered.
- Use this shape:

```markdown
- REQ-001: The system must ...
- REQ-002: The workflow must ...
- REQ-003: The user must be able to ...
```

## Acceptance Criteria

- After requirements are complete, default to generating acceptance criteria
  automatically unless the user explicitly says they want to provide or control
  the acceptance criteria themselves.
- Generate exactly one acceptance criterion for every requirement.
- Each acceptance criterion must correspond to the requirement with the same
  numeric index: `REQ-001` maps to `AC-001`, `REQ-002` maps to `AC-002`, and so
  on.
- Store acceptance criteria in the `Acceptance Criteria` custom field,
  `customfield_10104`.
- Format the field as a Markdown unordered list.
- Prefix every acceptance criterion with a stable `AC-###` identifier, matching
  the corresponding `REQ-###` number.
- Use this shape:

```markdown
- AC-001: Given ..., when ..., then ...
- AC-002: Given ..., when ..., then ...
- AC-003: Given ..., when ..., then ...
```

## Requirement Subtasks

- After requirements are locked, create one `Subtask` under the parent issue for
  every requirement.
- The subtask summary must use the same name as the requirement. Preserve the
  `REQ-###` prefix so the subtask maps clearly back to the requirement.
- Example: `REQ-001: Add settings page` creates a subtask named
  `REQ-001: Add settings page`.
- Populate each generated subtask's `Requirements` field (`customfield_10103`)
  with the one `REQ-###` requirement it represents.
- Populate each generated subtask's `Acceptance Criteria` field
  (`customfield_10104`) with the matching `AC-###` acceptance criterion.
- Keep each generated subtask description to a brief one-sentence explanation of
  the requirement work.
- Do not create duplicate subtasks if a matching requirement subtask already
  exists under the parent issue.
- Generate the subtasks after requirements are settled and before moving the
  parent issue out of `REQUIREMENTS`.

## Completion Rule

- Before moving an issue out of `REQUIREMENTS`, confirm that
  `customfield_10103` is populated with well-formed `REQ-###` requirements.
- Before moving an issue out of `REQUIREMENTS`, confirm that
  `customfield_10104` is populated with matching `AC-###` acceptance criteria
  for every requirement.
- Before moving an issue out of `REQUIREMENTS`, confirm that every requirement
  has a matching child `Subtask`.
- Do not leave `REQUIREMENTS` with only the rough `TO DO` overview when the issue
  type is `Story`, `Bug`, or `Task`.
- Once requirements, acceptance criteria, and requirement subtasks are complete,
  transition the main `Story`, `Bug`, or `Task` issue from `REQUIREMENTS` to
  `TECH LEAD`.
- Treat the `REQUIREMENTS -> TECH LEAD` transition as the normal handoff point
  from requirements gathering to technical review.
- If Jira does not expose a direct `TECH LEAD` transition, inspect live
  transitions and report the blocker instead of pretending the handoff happened.
